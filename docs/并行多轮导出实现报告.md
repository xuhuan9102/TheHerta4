# 并行多轮导出实现报告

## 概述

本次实现基于现有的多文件导出与形态键导出流程，以及已完成的并行前处理机制，新增了“并行多轮导出”能力。

设计目标如下：

1. 首轮导出保留在主线程执行。
2. 尾轮导出与后处理保留在主线程执行。
3. 中间所有仅生成 Buffer 的导出轮次，根据用户配置的实例数自动分发到多个独立 Blender 进程并行执行。
4. 保持与现有导出逻辑兼容，不改变首轮 INI 生成、形态键分类报告生成、多文件和形态键后处理的语义。
5. 保留未来扩展为“完整并行导出 worker”的能力。

## 实现结果

### 新增模块

新增文件：

1. [blueprint/export_parallel.py](blueprint/export_parallel.py)

该模块包含两层职责：

1. `ExportRoundExecutor`
   - 负责执行单轮导出。
   - 统一处理轮次状态设置、物体收集、前处理、蓝图解析、Buffer 导出、清理副本。
   - 可由主线程直接调用，也可由子进程 worker 调用。

2. `ParallelExportCoordinator`
   - 负责中间轮次任务拆分。
   - 负责创建快照、生成 manifest、启动 Blender 子进程池、收集结果。
   - worker 只执行纯 Buffer 导出，不执行后处理。

   - 将原先分支式的多轮串行导出改为“导出轮次计划驱动”。
   - 首轮与尾轮固定在主线程执行。
   - 中间轮次在启用配置后走并行调度。

2. [common/global_properties.py](common/global_properties.py)
   - 新增 `enable_parallel_export_rounds` 开关。
   - 复用现有 Blender 路径、实例数、超时和临时文件保留配置。

3. [ui/ui_panel_basic.py](ui/ui_panel_basic.py)
   - 主面板增加“启用并行多轮导出”开关。
   - 与并行前处理共用一组 Blender 子进程参数。

4. [blueprint/__init__.py](blueprint/__init__.py)
   - 注册 `export_parallel` 模块，确保主进程与子进程 worker 都可加载。

## 当前执行模型

### 轮次计划

在 [ui/ui_func_export.py](ui/ui_func_export.py) 中，导出会先被抽象为一个轮次计划数组。

不同模式下的计划如下：

1. 单次导出
   - 仅 1 轮。
   - 不使用并行。

2. 形态键导出
   - 第 1 轮：`Meshes0000`，主线程，生成 INI 和分类报告。
   - 中间轮：`Meshes1001` 到倒数第二个槽位，若存在，则进入并行队列。
   - 最后一轮：最后一个 `Meshes1xxx`，主线程执行。
   - 导出完成后，主线程执行后处理。

3. 多文件导出
   - 第 1 轮：`Meshes01`，主线程，生成 INI。
   - 中间轮：`Meshes02` 到倒数第二轮，若存在，则进入并行队列。
   - 最后一轮：最后一个 `MeshesNN`，主线程执行。
   - 导出完成后，主线程执行后处理。

4. 组合导出
   - 第 1 轮固定是形态键基态 `Meshes0000`，主线程执行。
   - 中间所有轮次包括：
     - 形态键中间槽位轮次
     - 多文件中间帧轮次
   - 上述轮次统一作为“中间轮次”进入并行任务池。
   - 最后一轮固定是整个导出计划的最后一轮，主线程执行。
   - 所有轮次完成后，主线程统一执行后处理。

## 为什么首轮和尾轮保留在主线程

### 首轮保留主线程的原因

首轮是整个导出链条的基准轮次，负责：

1. 生成主 INI。
2. 生成形态键分类报告。
3. 作为后续 Buffer 文件大小一致性校验的基准。

这些输出会被后续轮次与后处理直接依赖，因此不适合并发启动。

### 尾轮保留主线程的原因

尾轮之后立刻进入后处理。将尾轮保留在主线程的原因是：

1. 可以保证所有中间子进程已经结束。
2. 可以保证所有 Buffer 文件目录已经完整落盘。
3. 可以在主线程内直接拿到最后一轮的 `blueprint_model`，然后立刻执行现有 `execute_postprocess_nodes()`。

这样不需要改写后处理逻辑，也不需要在子进程和主进程间传递复杂的后处理上下文。

## 子进程 worker 的执行内容

中间轮次 worker 在 [blueprint/export_parallel.py](blueprint/export_parallel.py) 中执行，处理步骤如下：

1. 打开主线程导出完成首轮后生成的快照工程。
2. 根据 manifest 恢复轮次配置：
   - 当前 Buffer 文件夹名
   - 当前导出索引
   - 当前形态键模式
3. 收集物体。
4. 执行前处理。
5. 更新蓝图节点引用到 `_copy` 副本。
6. 解析蓝图模型。
7. 执行 `export_buffers_only()`。
8. 清理副本物体。
9. 把 Buffer 文件夹大小写入结果清单。

worker 不执行：

1. INI 生成
2. 分类报告生成
3. 后处理

## 与并行前处理的关系

### 主线程轮次

主线程首轮和尾轮在执行时，仍然可以继续使用已实现的并行前处理功能。

### 子进程轮次

子进程 worker 当前会强制关闭“嵌套并行前处理”，原因如下：

1. 避免导出轮次并行后，再在每个 worker 内部继续扩散进程数。
2. 避免系统资源失控。
3. 优先保证并行多轮导出的稳定性。

因此，当前版本采用的策略是：

1. 轮次级别允许并行。
2. worker 内部前处理保持串行。

后续如果需要，可以再引入更严格的资源配额控制后，允许“轮次并行 + 受控前处理并行”的组合模式。

## 当前配置项

主面板中新增或复用的相关配置如下：

1. `启用并行前处理`
2. `启用并行多轮导出`
3. `Blender 可执行文件`
4. `并行实例数`
5. `单任务超时(秒)`
6. `保留临时文件`

其中：

1. 并行前处理和并行多轮导出共用同一个 Blender 路径。
2. 二者共用同一个实例数上限。
3. 二者共用同一个超时设置。
4. 临时文件保留配置会同时影响并行前处理和并行导出。

## 稳定性策略

### 快照策略

中间轮次并行执行时，不直接操作当前主工程，而是：

1. 主线程先完成首轮导出。
2. 再创建一个 `.blend` 快照。
3. 所有中间 worker 都从这个快照启动。

这样可以保证：

1. 子进程环境与主线程导出逻辑尽量一致。
2. 每个子进程互不影响。
3. 不污染用户当前打开的工程。

### 文件冲突控制

中间轮次只允许写入各自独立的 Buffer 目录，例如：

1. `Meshes1001`
2. `Meshes1002`
3. `Meshes02`
4. `Meshes03`

由于每轮写入目录互不相同，因此在当前设计下可并行落盘。

### 一致性校验

每轮完成后会记录 Buffer 文件夹总大小。

主线程将用首轮结果作为基准，对所有中间轮次和尾轮进行一致性校验；如发现大小不一致，则直接报错并停止后处理。

## 当前限制

### 1. worker 只支持纯 Buffer 导出

这是刻意保守的设计。首轮和尾轮仍由主线程承担高耦合逻辑，以保证：

1. INI 生成语义不变。
2. 分类报告语义不变。
3. 后处理逻辑不需要重写。

### 2. 子进程当前不启用嵌套并行前处理

这不是功能缺失，而是稳定性权衡。

### 3. 未做 Blender 运行时实机回归

本次修改已经通过静态错误检查，但尚未在 Blender 内完成完整实机验证。

因此建议按以下顺序测试：

1. 多文件导出 3 轮场景
2. 形态键导出 3 到 4 轮场景
3. 组合导出场景
4. 并行前处理与并行多轮导出同时开启的场景

## 建议的验证步骤

### 验证 1：多文件导出

1. 打开并行多轮导出。
2. 设置并行实例数为 2。
3. 准备一个 4 轮多文件节点场景。
4. 观察是否满足：
   - `Meshes01` 主线程生成
   - 中间轮次并行生成
   - 最后一轮主线程生成
   - 后处理正常执行

### 验证 2：形态键导出

1. 准备至少 4 个形态键轮次的场景。
2. 检查是否满足：
   - `Meshes0000` 主线程生成
   - 中间 `Meshes1xxx` 并行生成
   - 最后一个 `Meshes1xxx` 主线程生成
   - `Shape_Key_Classification` 正常生成
   - 后处理读取成功

### 验证 3：组合导出

1. 同时存在形态键节点和多文件导出节点。
2. 验证中间轮次是否跨阶段统一并行执行。
3. 验证最终后处理是否仍按原顺序执行。

## 总结

本次实现已经把多轮导出从“每轮都必须串行主线程执行”改成了：

1. 主线程首轮导出
2. 子进程并行中间轮次导出
3. 主线程尾轮导出
4. 主线程后处理

它保留了当前插件最关键的稳定性边界，同时建立了一个清晰的并行轮次调度框架，为后续继续扩展成“更完整的子进程导出”打下了结构基础。
