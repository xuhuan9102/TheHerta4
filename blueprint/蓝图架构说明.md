## 一、架构总览

TheHerta4 蓝图系统是一个基于 Blender 自定义节点编辑器的 **Mod 逻辑可视化编排框架**。用户通过在节点编辑器中拖拽连线，定义物体从原始状态到最终导出的完整处理路径。系统自动解析节点连接关系，生成处理链（ProcessingChain），并驱动多游戏格式的 Mod 导出。

### 两种数据流

蓝图系统中存在两种独立的数据流，通过不同颜色的 Socket 区分：

| 数据流 | Socket 类型 | 颜色 | 用途 |
|--------|-------------|------|------|
| **青色数据流** | `SSMTSocketObject` | `(0.0, 0.8, 0.8)` | 物体处理链路：从 Object_Info 出发，经过中间处理节点，到达 Result_Output |
| **橙色数据流** | `SSMTSocketPostProcess` | `(1.0, 0.5, 0.0)` | 后处理链路：从 Result_Output 的输出端出发，串联各后处理节点 |

两种数据流在 **Result_Output（Mod输出）** 节点汇合——青色流进入其输入端，橙色流从其输出端引出。

## 二、模块分类

### 2.1 核心处理

核心处理模块负责蓝图系统的底层架构：节点树定义、数据模型、链路遍历、导出调度。它们不对应任何用户可见的节点，但支撑着整个系统的运转。

| 文件 | 核心类 | 功能说明 |
|------|--------|----------|
| `node_base.py` | `SSMTBlueprintTree`, `SSMTNodeBase`, `SSMTSocketObject`, `SSMTSocketPostProcess` | 节点树类型定义、所有节点的基类、两种 Socket 类型、自动宽度计算 |
| `__init__.py` | — | 模块注册表，按 required/optional 分级注册所有子模块 |
| `model.py` | `ProcessingChain`, `ChainGroup`, `BluePrintModel` | 蓝图数据模型：处理链数据结构、链路合并、重命名执行、跨IB处理、后处理调度 |
| `chain_traverser.py` | `ChainTraverser` | 正向链路遍历器：从 Object_Info 出发递归遍历节点图，生成 ProcessingChain 列表，处理分叉/去重/嵌套 |
| `export_helper.py` | `BlueprintExportHelper` | 导出辅助：蓝图树查找、节点查找、连接分析、多文件导出管理、形态键分类报告 |
| `export_parallel.py` | `ExportRoundExecutor`, `ParallelExportCoordinator` | 导出执行器：单轮导出流程（前处理→解析→导出）；并行导出协调（多 Blender 子进程） |

### 2.2 辅助工具

辅助工具模块提供节点编辑器的交互增强功能，不参与数据处理链路。

| 文件 | 核心类/功能 | 说明 |
|------|-------------|------|
| `sync.py` | `SSMT_OT_ToggleSync` 等 7 个 Operator | 节点-物体双向同步：定时器轮询 + msgbus + depsgraph，防循环机制，物体重命名检测 |
| `node_menu.py` | 6 个分类菜单 + 4 个右键菜单 Operator + 对齐/批量连接工具 | 节点添加菜单系统、上下文菜单快捷操作 |
| `node_preset.py` | `NodePresetManager`, `SSMT_OT_SaveNodePreset`, `SSMT_OT_LoadNodePreset` | 节点预设保存/加载，JSON 序列化 |
| `nest_navigate.py` | `SSMT_OT_BlueprintNestNavigate`, `SSMT_OT_CreateBlueprintFromNest` | 嵌套蓝图 Tab 键导航、从嵌套节点创建新蓝图 |
| `shader_quick_connect.py` | QuickTransparent / QuickEmission / QuickDiffuse / QuickPrincipled | Shader 编辑器快速连接工具（**不属于蓝图系统**，是独立辅助工具） |

### 2.3 前处理

前处理模块在导出前对 Blender 物体执行一系列预处理操作，确保几何数据符合导出要求。

| 文件 | 核心类 | 功能说明 |
|------|--------|----------|
| `preprocess.py` | `PreProcessHelper` | 前处理管线：创建副本 → 应用约束 → 应用修改器 → 三角化 → 应用变换 → 应用形态键 → 非镜像恢复；更新节点引用指向副本；导出后清理副本 |
| `preprocess_cache.py` | `PreProcessCache` | 前处理缓存：SHA-256 哈希计算（含顶点/UV/形态键/修改器/约束/顶点组），缓存命中时直接加载 .blend 文件跳过前处理 |
| `preprocess_parallel.py` | `ParallelPreprocessCoordinator` | 并行前处理：按权重分组物体 → 保存快照 → 启动多个 Blender 子进程 → 整合结果 |

### 2.4 导出流程相关节点（青色数据流）

这些节点通过 `SSMTSocketObject`（青色）连接，构成从物体输入到 Mod 输出的主处理链路。数据流方向为：

```
Object_Info → [中间处理节点...] → Result_Output
```

| 文件 | 节点类名 | 中文名 | 在链路中的角色 | 功能说明 |
|------|----------|--------|----------------|----------|
| `node_obj.py` | `SSMTNode_Object_Info` | 物体信息 | **链路起点** | 引用一个 Blender 物体，含 object_name/draw_ib/index_count/first_index/component/alias_name |
| `node_obj.py` | `SSMTNode_Object_Group` | 物体组 | 聚合器 | 将多个 Object_Info 聚合为一组，动态输入槽 |
| `node_shapekey.py` | `SSMTNode_ShapeKey` | 形态键 | 参数注入 | 为链路注入一个形态键参数（名称/虚拟键码/注释） |
| `node_shapekey.py` | `SSMTNode_ShapeKey_Output` | 形态键输出 | 参数聚合 | 聚合多个 ShapeKey 节点的参数 |
| `node_rename.py` | `SSMTNode_Object_Rename` | 重命名物体 | 名称变换 | 多规则重命名（正则搜索/替换），支持反向映射 |
| `node_swap.py` | `SSMTNode_ObjectSwap` | 物体切换 | 链路分叉 | 支持 cycle/toggle/hold 切换类型，一个输入分叉为多个输出选项 |
| `node_cross_ib.py` | `SSMTNode_CrossIB` | 跨IB | IB映射 | 定义源IB到目标IB的映射，支持 IndexCount 匹配和 IB Hash 匹配 |
| `node_datatype.py` | `SSMTNode_DataType` | 数据类型替换 | 格式覆盖 | 加载 JSON 配置覆盖 D3D11 顶点声明格式 |
| `node_vertex_group_match.py` | `SSMTNode_VertexGroupMatch` | 顶点组匹配 | 权重映射 | KD-Tree + Chamfer 距离算法，将源物体顶点组映射到目标物体 |
| `node_vertex_group_process.py` | `SSMTNode_VertexGroupProcess` | 顶点组处理 | 权重处理 | 接收映射输入，执行重命名/合并/清理/填充/排序 |
| `node_vertex_group_mapping_input.py` | `SSMTNode_VertexGroupMappingInput` | 映射表输入 | 映射来源 | 引用文本块作为顶点组映射输入 |
| `node_nest.py` | `SSMTNode_Blueprint_Nest` | 蓝图嵌套 | 子图引用 | 引用另一个蓝图节点组，到达子图 Result_Output 后跳回父图继续遍历 |
| `node_obj.py` | `SSMTNode_Result_Output` | Mod输出 | **链路终点** | 青色流的汇聚点，含全局导出设置（顶点去重、游戏特定选项）；橙色流的起点 |

### 2.5 后处理相关节点（橙色数据流）

这些节点通过 `SSMTSocketPostProcess`（橙色）连接，在主导出完成后对输出文件进行二次处理。数据流方向为：

```
Result_Output → [后处理节点串联...] → 最终输出
```

所有后处理节点继承自 `SSMTNode_PostProcess_Base`（定义在 `node_postprocess_base.py`），必须实现 `execute_postprocess(mod_export_path)` 方法。

| 文件 | 节点类名 | 中文名 | 功能说明 |
|------|----------|--------|----------|
| `node_postprocess_base.py` | `SSMTNode_PostProcess_Base` | 后处理基类 | 抽象基类，定义 execute_postprocess() 接口和备份工具 |
| `node_postprocess_buffer_cleanup.py` | `SSMTNode_PostProcess_BufferCleanup` | 缓冲区清理 | 扫描并删除未被任何 INI 文件引用的 .buf 文件 |
| `node_postprocess_health.py` | `SSMTNode_PostProcess_HealthDetection` | 血量检测 | 追加血量检测模块（仅限 ZZZ），自定义角色哈希和血量映射级数 |
| `node_postprocess_material.py` | `SSMTNode_PostProcess_Material` | 材质转资源 | 根据场景物体材质和纹理创建资源引用，支持名称映射和材质切换变量 |
| `node_postprocess_multifile.py` | `SSMTNode_PostProcess_MultiFile` | 多文件配置 | 为指定哈希值物体生成多文件动画配置，紧凑缓冲区和顶点增量存储 |
| `node_postprocess_resource_merge.py` | `SSMTNode_PostProcess_ResourceMerge` | 资源合并 | MD5 哈希去重，合并相同内容的资源引用并删除重复贴图文件 |
| `node_postprocess_shapekey.py` | `SSMTNode_PostProcess_ShapeKey` | 形态键配置 | 读取分类文本生成多形态叠加混合 INI 配置，支持紧凑缓冲区/顶点增量/优化查找 |
| `node_postprocess_slider.py` | `SSMTNode_PostProcess_SliderPanel` | 滑块面板 | 扫描 INI 中形态键强度参数，生成可交互滑块 UI |
| `node_postprocess_vertex_attrs.py` | `SSMTNode_PostProcess_VertexAttrs` | 顶点属性定义 | 为形态键/多文件配置提供顶点属性定义（float/float2/.../double4） |
| `node_cross_ib.py` | `SSMTNode_PostProcess_CrossIB` | 跨IB后处理 | 复制跨IB所需的 HLSL 文件到资源目录 |

> **注意**：`SSMTNode_PostProcess_CrossIB` 虽然是后处理节点（使用橙色 Socket），但其定义与 `SSMTNode_CrossIB` 共享 `node_cross_ib.py` 文件。

## 三、完整数据流执行流程

以下假设所有节点都被连入节点树中（除多文件导出和形态键配置后处理节点），描述从用户触发导出到最终输出的完整流程。

### 3.1 阶段一：前处理（PreProcess）

**执行者**：`PreProcessHelper` / `ParallelPreprocessCoordinator`

**目的**：将 Blender 场景中的原始物体处理为符合导出要求的几何数据，同时保留原始物体不被修改。

```
原始物体 ──复制──→ 副本物体(_copy) ──处理──→ 可导出的副本
```

**步骤**：

1. **收集物体名称**：遍历蓝图树中所有未静音的 Object_Info 节点和嵌套蓝图中的 Object_Info 节点，收集物体名称
2. **缓存检查**（如启用）：计算每个物体的 SHA-256 哈希值（含顶点/UV/形态键/修改器/约束/顶点组），检查缓存是否命中
3. **创建副本**：为每个物体创建 `_copy` 后缀的副本（含网格数据复制）
4. **应用约束**：将物体上的 Blender 约束（如骨骼绑定）烘焙到副本
5. **应用修改器**：将修改器（如镜像、细分）应用到副本；有形态键的物体使用特殊处理流程
6. **三角化**：将四边面/多边面转换为三角面
7. **应用变换**：将 Location/Rotation/Scale 烘焙到顶点数据
8. **应用形态键**：将当前形态键混合结果烘焙到网格，然后删除形态键
9. **非镜像工作流恢复**（如启用）：恢复非镜像物体的完整几何
10. **更新节点引用**：将蓝图树中 Object_Info 和 MultiFile_Export 节点的 `object_name` 指向副本

### 3.2 阶段二：蓝图解析（Blueprint Parsing）

**执行者**：`BluePrintModel.__init__()`

**目的**：构建蓝图数据模型，收集所有节点信息。

**步骤**：

1. **获取蓝图树**：从 Context / Runtime / GlobalConfig 三级回退查找当前 `SSMTBlueprintTreeType` 节点树
2. **查找输出节点**：找到 `SSMTNode_Result_Output` 节点，若无则报错
3. **收集后处理节点**：从 Result_Output 的橙色输出端出发，遍历所有连接的后处理节点（跳过静音节点）
4. **收集特殊节点**：遍历整棵树，收集顶点组处理节点、多文件导出节点、嵌套蓝图、跨IB节点
5. **解析嵌套蓝图**：递归解析 Blueprint_Nest 引用的子蓝图树（检测循环引用）

### 3.3 阶段三：链路遍历与处理链生成

**执行者**：`ChainTraverser.traverse_all_chains()`

**目的**：从每个 Object_Info 出发，正向遍历节点连接图，生成完整的 ProcessingChain 列表。

**步骤**：

1. **收集起点节点**：所有未静音的 Object_Info 节点 + 嵌套蓝图中的 Object_Info 节点
2. **逐个遍历**：对每个起点节点调用 `_traverse_forward()`
3. **节点处理逻辑**（按节点类型）：

   | 节点类型 | 遍历行为 |
   |----------|----------|
   | Object_Info | 起点记录物体名称，继续向前 |
   | Object_Group | 记录分组名到 group_stack，继续向前 |
   | ShapeKey | 提取形态键参数（M_Key），追加到 shapekey_params，继续向前 |
   | ShapeKey_Output | 记录到节点路径，继续向前 |
   | Object_Rename | 预执行重命名，记录重命名历史和签名，继续向前 |
   | ObjectSwap | 记录输出槽索引到 swap_node_option_values，触发链路分叉 |
   | DataType | 记录到节点路径，继续向前 |
   | VertexGroupMatch | 记录到 vertex_group_mapping_nodes，继续向前 |
   | VertexGroupMappingInput | 记录到 vertex_group_mapping_nodes，继续向前 |
   | VertexGroupProcess | 记录到 vertex_group_process_nodes，继续向前 |
   | CrossIB | 记录到节点路径，继续向前 |
   | Blueprint_Nest | 记录到节点路径，进入子蓝图继续遍历 |
   | Result_Output（子图） | 标记到达子图输出，跳回父图的 Nest 节点继续 |
   | Result_Output（主图） | 标记 `reached_output = True`，链路完成 |

4. **链路分叉处理**：当节点有多个输出连接时，深拷贝当前链路并为每个分支创建独立副本（同时复制 Blender 物体避免冲突）
5. **去重**：基于 `object_name + chain_hash` 去除完全重复的链路
6. **同名物体去重**：同一物体被多条链路引用时，自动创建 `_dup` 后缀的独立副本

### 3.4 阶段四：链路执行（重命名 + 顶点组处理）

**执行者**：`BluePrintModel._execute_chain_nodes_sequentially()`

**目的**：按照处理链中节点的顺序，依次执行重命名和顶点组处理操作。

**步骤**：

1. **筛选有效链路**：仅处理 `is_valid=True` 且 `reached_output=True` 的链路
2. **按节点路径顺序执行**：对每条链路中的每个节点：
   - **Object_Rename**：调用 `SSMTNode_Object_Rename.apply_to_object_name()` 执行重命名，更新物体名称
   - **VertexGroupProcess**：调用 `node.process_object(obj)` 执行顶点组操作（重命名/合并/清理/填充/排序）
3. **记录名称映射**：重命名后的 `旧名 → 新名` 映射保存到 `BluePrintModel._object_name_mapping`，供后续链路使用

### 3.5 阶段五：ObjectSwap 集成

**执行者**：`integrate_object_swap_to_blueprint_model()`（来自 `node_swap_processor.py`）

**目的**：将 ObjectSwap 节点的切换逻辑集成到蓝图模型中，生成 M_Key 条件。

**步骤**：

1. **SwapKeyRegistry**：为每个 ObjectSwap 节点分配唯一的 swapkey 索引
2. **ObjectSwapChainProcessor**：遍历处理链，为包含 ObjectSwap 的链路构建 M_Key 条件（基于 swap_node_option_values）
3. **将 M_Key 条件注入**到对应处理链的 shapekey_params 中

### 3.6 阶段六：跨IB处理

**执行者**：`BluePrintModel._process_cross_ib_nodes()`

**目的**：处理跨 Index Buffer 映射，建立源IB到目标IB的对应关系。

**步骤**：

1. **收集映射信息**：从每个 CrossIB 节点提取 IB 映射字典和匹配模式
2. **检测跨IB重命名映射**：如果链路同时包含 CrossIB 和 Object_Rename，从重命名历史中提取 IndexCount/IBHash 映射规则，自动应用到 CrossIB 节点
3. **标记跨IB物体**：遍历有效链路，将经过 CrossIB 节点的物体标记为跨IB物体
4. **构建 VB 条件映射**：记录每个源→目标映射的 VertexShader 条件

### 3.7 阶段七：DrawCallModel 构建

**执行者**：`BluePrintModel._build_draw_call_models_from_chains()`

**目的**：将处理链转换为导出引擎可用的 DrawCallModel 列表。

**步骤**：

1. **筛选有效链路**
2. **对每条链路调用 `chain.to_draw_call_model()`**：
   - 构建虚拟物体名称（含前缀信息：draw_ib/index_count/first_index/component）
   - 设置源物体名称
   - 深拷贝形态键参数列表到 `work_key_list`
3. **输出调试信息**到 Blender 文本编辑器（"物体处理链"文本块）

### 3.8 阶段八：游戏格式导出

**执行者**：`ExportRoundExecutor.export_with_ini()` / `export_buffers_only()`

**目的**：根据当前选择的游戏格式，将 DrawCallModel 列表导出为对应的 Mod 文件。

**步骤**：

1. **选择导出器**：根据 `GlobalConfig.logic_name` 选择对应的导出器（EFMI/GIMI/HIMI/ZZMI/WWMI/SRMI 等）
2. **导出 Buffer 文件**：将几何数据（顶点/索引）写入 .buf 文件
3. **生成 INI 配置**（如启用）：生成 3DMigoto/XXMI 格式的 INI 配置文件，包含：
   - ShaderOverride 段（匹配游戏的 Draw Call）
   - TextureOverride 段（替换 Buffer 引用）
   - CommandList 段（组织绘制逻辑）
   - KeySwap 段（物体切换逻辑，来自 ObjectSwap 节点）
   - Constants 段（常量定义）
   - Resource 段（材质/纹理引用，来自 Material 后处理节点）

### 3.9 阶段九：后处理节点执行

**执行者**：`BluePrintModel.execute_postprocess_nodes()`

**目的**：在主导出完成后，按顺序执行后处理节点对输出文件进行二次处理。

**执行顺序**：按后处理链路的连接顺序依次执行。

**各后处理节点的执行内容**：

| 执行顺序 | 后处理节点 | 执行内容 |
|----------|-----------|----------|
| 1 | 顶点属性定义 | 提供顶点属性定义数据，供后续节点使用（不直接修改文件） |
| 2 | 材质转资源 | 扫描场景物体材质/纹理，在 INI 中生成 Resource 引用段 |
| 3 | 跨IB后处理 | 复制跨IB所需的 HLSL 文件到资源目录 |
| 4 | 血量检测 | 在 INI 末尾追加血量检测模块（ZZZ 专用） |
| 5 | 滑块面板 | 扫描 INI 中形态键强度参数，生成滑块 UI 配置 |
| 6 | 资源合并 | MD5 去重，合并相同内容的资源引用，删除重复贴图 |
| 7 | 缓冲区清理 | 扫描并删除未被 INI 引用的 .buf 文件 |

> **注意**：实际执行顺序取决于节点在橙色链路中的连接顺序。上表为典型推荐顺序。

### 3.10 阶段十：清理与恢复

**执行者**：`PreProcessHelper.cleanup_copies()`

**目的**：删除前处理阶段创建的副本物体，恢复蓝图节点引用到原始物体。

**步骤**：

1. **删除副本物体**：删除所有 `_copy` 后缀的 Blender 物体
2. **恢复节点引用**：将 Object_Info 和 MultiFile_Export 节点的 `object_name` 从副本名恢复为原始名
3. **清空运行时状态**：清空 `original_to_copy_map`、`created_copies`、`modified_nodes`
